# FUSE Migration: 0.9.x -> 1.0.0

This guide covers what to expect when moving from any FUSE `0.9.x` release to `1.0.0`.

## No source migration required

No language or runtime source changes are required to move from `0.9.x` to `1.0.0`.

All `0.9.1`–`0.9.10` releases were additive. Every feature introduced in that window
(`http.*` client, asset imports, typed query forms, `fuse deps lock`, `component` declarations,
`aria-*` validation) extends the `0.9.0` baseline without removing or redefining existing syntax
or runtime behavior.

## If you are coming from 0.8.x

Apply the `0.8.x → 0.9.0` changes first, then proceed to `1.0.0`:

- See `guides/migrations/0.8-to-0.9.md` for the required HTML attr syntax migration.

The summary: replace map-literal HTML attrs with named attrs and remove commas between HTML attrs.
After that, no additional source changes are needed for `1.0.0`.

## Tooling notes

### `requires network` gates `http.*` client calls (since 0.9.6)

If your module calls `http.request`, `http.get`, or `http.post`, it must declare `requires network`.
This was not enforced before `0.9.6`.

```fuse
requires network

fn fetch(url: String) -> String!http.error:
  let resp = http.get(url) ?!
  return resp.body
```

### Non-`2xx` HTTP responses are errors (since 0.9.6)

`http.get` / `http.post` / `http.request` return `Err(http.error)` for non-`2xx` responses
(code `"http_status"`). Earlier code that treated all responses as `Ok` must be updated.

```fuse
# Before (0.9.5 and earlier — no HTTP client existed)
# After (0.9.6+):
match http.get(url):
  Ok(resp) -> resp.body
  Err(e)   -> log("error", "request failed", e.code)
```

### Dependency lockfile (since 0.9.3)

Multi-package projects should commit a `fuse.lock` file and use `fuse deps lock` to keep it
current. CI can enforce deterministic resolution with `fuse deps lock --check`.

```bash
fuse deps lock                   # refresh lockfile
fuse deps lock --check           # verify it matches resolved graph
fuse check --frozen              # fail fast if lockfile is stale
```

### Guide docs are hand-maintained (since 0.9.10)

`scripts/generate_guide_docs.sh` has been removed. Update `guides/` files directly in the same
PR as any language or runtime change.

## Stability guarantee at 1.0.0

`1.0.0` freezes the language contract documented in `spec/fls.md` and `spec/runtime.md`.
Future `1.x.y` releases will not introduce breaking changes to:

- syntax and grammar
- type-system rules
- runtime boundary behavior (validation, JSON, config, HTTP, DB)
- builtin signatures
- diagnostic code names

Breaking changes, if ever required, will be introduced in a new major version with a migration
guide following this same format.
