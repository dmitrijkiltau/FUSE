# 0.9.8 plan: outbound HTTP production completion

## Release status

- Decision: plan as the next patch after `0.9.7`, not in parallel scope.
- Objective: close the most visible production gap in the `http.*` surface before `1.0.0`.
- Current progress: M1-M5 complete; `0.9.8` outbound HTTP production work is release-ready.

| Milestone | Status | Progress | Notes |
|---|---|---:|---|
| M1 — HTTPS contract freeze | Complete | 100% | Contract frozen below: no signature changes, validated TLS by default, manual redirects, and stable error-code mapping for implementation milestones. |
| M2 — AST runtime HTTPS support | Complete | 100% | AST outbound client now supports validated `https://` requests, preserves request/response shapes, and has mixed HTTP/HTTPS regression coverage. |
| M3 — Native parity and diagnostics | Complete | 100% | Native hostcalls now exercise the same HTTPS transport path as AST, with parity coverage for success, non-`2xx`, and TLS diagnostic surfaces. |
| M4 — Timeout, header, and observability polish | Complete | 100% | Timeout messages now identify the failing phase/deadline, reserved request-owned headers are regression-covered, and outbound client logs/metrics align across AST/native backends. |
| M5 — Docs, examples, and release validation | Complete | 100% | README/spec/reference docs now describe the HTTPS contract, LSP help text is aligned, a new HTTPS example is checked in, and focused regression/example validation passed. |

## Why `0.9.8`

`0.9.6` intentionally shipped an HTTP client that only supports `http://`. That was a good patch-sized
first step, but it leaves a visible pre-`1.0.0` gap for real deployments. The next patch should
finish the transport surface rather than widen it.

## Scope

### In scope

- `https://` support for `http.request`, `http.get`, and `http.post`
- clear certificate/handshake/timeout failure mapping into `http.error`
- AST/native parity for all supported schemes
- doc and example coverage for outbound HTTPS usage

### Out of scope

- streaming request/response bodies
- redirect policy expansion unless needed for baseline interoperability
- broader HTTP verb additions unless already trivial under the same surface
- any source-breaking changes to the existing `http.*` API

## Milestones

### M1 — HTTPS contract freeze

Goal: define the stable pre-`1.0.0` HTTPS contract.

Status: complete.

Deliverables:

- Specify certificate validation posture.
- Specify error mapping for DNS/TLS/connect/timeout/HTTP-status failures.
- Decide whether redirects remain manual in `0.9.x`.
- Confirm no signature changes are needed for the current `http.*` builtins.

Exit criteria:

- spec wording can be updated without ambiguity
- no patch-breaking API changes are required

Frozen decisions for `0.9.8` implementation work:

#### 1. API surface remains unchanged

- `http.request`, `http.get`, and `http.post` keep their current signatures.
- `http.response` and `http.error` keep their current field sets and field names.
- HTTPS support is additive only: existing `http://` behavior remains valid and source-compatible.

#### 2. Supported schemes and URL posture

- `0.9.8` must support both `http://` and `https://` URLs for `http.request/get/post`.
- Any other scheme remains rejected before I/O begins.
- Malformed URLs remain `invalid_url` failures.
- Unsupported but well-formed schemes (for example `ftp://` or `file://`) must resolve to
  `Err(http.error)` with `code = "unsupported_scheme"`.
- This milestone freezes the target contract only; current `0.9.7` implementation still rejects
  `https://` until M2 lands.

#### 3. TLS and certificate-validation posture

- HTTPS requests must perform certificate-chain validation against the platform/system trust store.
- HTTPS requests must perform hostname verification against the request URL host.
- Expired, hostname-mismatched, untrusted-root, and otherwise invalid peer certificates must fail
  closed.
- `0.9.x` does not add an insecure skip-verify mode, custom CA bundle parameter, or client
  certificate surface.
- Plain `http://` remains permitted when explicitly requested by user code; there is no automatic
  scheme upgrade or HSTS-like policy in `0.9.x`.

#### 4. Frozen `http.error.code` mapping

Observable failure classes for `0.9.8` are:

| Failure class | `http.error.code` | Contract notes |
|---|---|---|
| malformed URL / invalid timeout / invalid header / invalid method | `invalid_request` or `invalid_url` | Preserve current request-validation split. |
| unsupported scheme | `unsupported_scheme` | Used for non-`http`/`https` schemes. |
| DNS resolution failure | `network_error` | Message should make clear that name resolution failed. |
| TCP connect failure | `network_error` | Includes refused/unreachable/reset-before-HTTP cases. |
| TLS handshake or certificate validation failure | `tls_error` | New explicit transport class for HTTPS-only failures. |
| timeout during connect / handshake / write / read | `timeout` | Timeout keeps priority over lower-level transport classification when the observable failure is a deadline expiry. |
| malformed HTTP response bytes | `invalid_response` | Shared across `http://` and `https://`. |
| non-`2xx` HTTP response | `http_status` | Preserve current behavior and response/error shapes. |

Additional mapping rules:

- `http.error.status` is populated only for `http_status` failures and for malformed responses that
  already exposed a parseable status line.
- `http.error.headers` and `http.error.body` remain populated only when response bytes were
  successfully received and parsed far enough to expose them.
- `http.error.message` remains implementation-produced text and may vary in exact phrasing, but it
  must clearly identify the method, URL, and failure class.

#### 5. Redirect policy stays manual in `0.9.x`

- `0.9.8` does not add automatic redirect following.
- `3xx` responses remain observable as ordinary non-`2xx` HTTP responses and therefore surface as
  `Err(http.error)` with `code = "http_status"`.
- Callers that want redirect behavior must inspect `status`/`headers["location"]` and issue a
  follow-up request explicitly.

#### 6. Existing request-shaping rules stay in force

- `timeout_ms` remains the only timeout control; default is still `30000`, `0` still disables the
  timeout, and negative values remain invalid requests.
- Header normalization remains lowercase.
- Runtime-owned request headers remain `host`, `connection`, and `content-length`; user code
  cannot override them.
- No streaming request/response bodies, redirect-policy flags, or broader verb additions are added
  under this milestone.

#### 7. Implementation acceptance bar for later milestones

- AST and native backends must expose the same supported schemes and the same error-code mapping.
- LSP/help text, README/spec/reference docs, and examples must use this frozen contract once M2-M5
  land.
- Any implementation detail that cannot satisfy this contract is a blocker for `0.9.8`, not a
  reason to widen or break the `http.*` API.

### M2 — AST runtime HTTPS support

Goal: implement working `https://` requests in the interpreter runtime.

Status: complete.

Deliverables:

- TLS-enabled request path
- preserved header normalization and timeout behavior
- unchanged `http.response` / `http.error` shapes

Exit criteria:

- AST backend passes HTTP/HTTPS mixed regression coverage

Implementation notes captured by this milestone:

- outbound client URL parsing now accepts both `http://` and `https://`, with default ports `80`
  and `443` respectively
- HTTPS requests use a validating TLS client path with hostname verification and system trust
  roots, while test coverage can inject an additional PEM root via process environment for local
  regression servers
- request header normalization, reserved-header rejection, timeout handling, and
  `http.response`/`http.error` shapes remain unchanged
- mixed regression coverage now exercises:
  - HTTP success / HTTP non-`2xx` behavior across AST/native execution
  - HTTPS handshake failure mapping to `tls_error`
  - AST HTTPS success against a local validating TLS server

### M3 — Native parity and diagnostics

Goal: ensure native behavior is indistinguishable from AST behavior.

Status: complete.

Deliverables:

- native runtime HTTPS parity
- matched error code/message rules where observable
- parity tests for success and failure cases

Exit criteria:

- no backend divergence on supported transport cases

Implementation notes captured by this milestone:

- native outbound `http.request/get/post` hostcalls continue to route through the shared
  `http_client` implementation, so HTTPS transport semantics are now shared across AST and native
  execution paths
- parity coverage now verifies:
  - HTTPS success across AST/native using the same injected local trust root
  - HTTPS handshake failure parity across AST/native
  - observable TLS diagnostic-message parity, normalized only for dynamic loopback port text
- no backend-specific divergence remains for supported scheme handling, `http_status`, or
  `tls_error` result shaping in the tested surface

### M4 — Timeout, header, and observability polish

Goal: make the HTTP client operationally safe enough for `1.0.0`.

Status: complete.

Deliverables:

- better timeout diagnostics
- deterministic handling for reserved/request-owned headers
- request log / metrics consistency for outbound failure classes where applicable

Exit criteria:

- no unresolved production-facing ambiguity remains in the transport contract

Implementation notes captured by this milestone:

- timeout failures now distinguish the transport phase in `http.error.message`
  (for example connect/read/write/TLS handshake), and timed failures include the configured
  deadline in milliseconds when one is present
- reserved request-owned header rejection remains case-insensitive and deterministic for the
  current runtime-owned set (`host`, `connection`, `content-length`), with explicit parity
  regression coverage
- outbound client requests now participate in the existing structured log / metrics hook surfaces:
  - structured logs emit `event = "http.client.request"`
  - metrics emit `metric = "http.client.request"`
  - both surfaces carry runtime, method, URL, status, outcome, response-bytes, and error-code
    information where applicable
- AST/native regressions now cover timeout diagnostics, reserved-header rejection, HTTPS success,
  TLS failure parity, and outbound observability emission

### M5 — Docs, examples, and release validation

Goal: ship the feature with the same contract quality as the rest of `0.9.x`.

Status: complete.

Deliverables:

- doc updates in `README.md`, `spec/runtime.md`, and generated reference docs
- at least one HTTPS example
- release smoke/regression coverage updates

Exit criteria:

- HTTPS support is documented, tested, and migration-neutral

Implementation notes captured by this milestone:

- `README.md`, `spec/runtime.md`, and generated `guides/reference.md` now describe the production
  outbound client contract: validated `https://` support, stable failure-code mapping, reserved
  request headers, manual redirects, and outbound observability hooks
- LSP signature help for `http.request`, `http.get`, and `http.post` now advertises both
  `http://` and validated `https://` support
- a new checked example, `examples/http_client_https.fuse`, demonstrates explicit `Ok` / `Err`
  handling for outbound HTTPS requests with configurable URL and timeout inputs
- release validation for this milestone included:
  - `./scripts/generate_guide_docs.sh`
  - `./scripts/cargo_env.sh cargo test -p fusec --test html_runtime --test parity_ast_native --test native_http_smoke`
