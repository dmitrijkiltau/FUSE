# 0.9.10 plan: 1.0 runway hardening

## Release status

- Decision: treat as the final planned `0.9.x` runway release before a `1.0.0` go/no-go call.
- Objective: prove that the documented contract, runtime parity, and release process are ready for
  a stable major release.
- Current progress: M1 complete.

| Milestone | Status | Progress | Notes |
|---|---|---:|---|
| M1 — 1.0 readiness scorecard | Complete | 100% | Scorecard artifact at `governance/scorecard-1.0.md`. |
| M2 — Parity and regression expansion | Not started | 0% | Increase confidence on AST/native/LSP/release-path parity. |
| M3 — Production dress rehearsal | Not started | 0% | Exercise build/package/run/deploy flows like a real `1.0.0` candidate. |
| M4 — Docs and migration freeze | Not started | 0% | Lock messaging and migration guidance for the major release. |
| M5 — 1.0 go/no-go decision | Not started | 0% | Decide whether to tag `1.0.0` or schedule another `0.9.x` hardening pass. |

## Why `0.9.10`

The last release before `1.0.0` should prioritize proof over breadth. By this point, the question
should not be “what else can we add?” but “can we defend the stable contract in production and in
tooling?”

## Scope

### In scope

- readiness scorecard across language/runtime/tooling/release ops
- expanded parity and regression coverage for stabilized surfaces
- production-style validation of CLI, AOT, packaging, docs, and reference-service workflows
- final documentation and migration freeze inputs for `1.0.0`

### Out of scope

- new major user-facing features unless they are strict blockers for `1.0.0`
- speculative post-`1.0` design work
- breaking contract changes

## Milestones

### M1 — 1.0 readiness scorecard

Goal: define a concrete checklist for stable-release quality.

Deliverables:

- scorecard covering semantics, runtime parity, LSP quality, docs, release automation, and examples
- explicit pass/fail thresholds where possible

Exit criteria:

- readiness can be reviewed as an artifact, not a conversation

### M2 — Parity and regression expansion

Goal: raise confidence that no known contract-facing divergence remains.

Deliverables:

- additional authority/parity coverage where current gaps exist
- release smoke and regression updates for newly stabilized `0.9.x` features
- performance/latency gate review for the stabilized line

Exit criteria:

- no known critical parity gaps remain open

### M3 — Production dress rehearsal

Goal: validate the workflows users will actually run at `1.0.0`.

Deliverables:

- full release packaging dry run
- reference-service deployment/recovery rehearsal
- verification that docs, artifacts, and scripts align with observed behavior

Exit criteria:

- release operators can execute the major-release path without ad hoc fixes

### M4 — Docs and migration freeze

Goal: stop contract churn and prepare stable messaging.

Deliverables:

- final pre-`1.0.0` doc audit
- migration guidance from the active `0.9.x` line to `1.0.0` if needed
- changelog/release-note structure prepared for stable-release expectations

Exit criteria:

- user-facing documentation is ready for a stability claim

### M5 — 1.0 go/no-go decision

Goal: make an explicit release call.

Deliverables:

- written decision: tag `1.0.0` or schedule another `0.9.x` hardening release
- if no-go, a bounded blocker list only

Exit criteria:

- there is a documented path from `0.9.10` to the actual major release
