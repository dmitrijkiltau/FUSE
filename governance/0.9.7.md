# 0.9.7 plan: markdown and JSON imports

## Release status

- Decision: Do not release `0.9.7` yet.
- Reason: the feature is still at planning stage, and `0.9.x` patch releases must remain
  non-breaking, fully documented, and parity-safe across AST/native/LSP.
- Current progress: planning drafted; implementation not started.

| Milestone | Status | Progress | Notes |
|---|---|---:|---|
| M1 — Semantics and surface freeze | In progress | 25% | Contract draft exists in this document; not yet implemented or ratified in specs/tests. |
| M2 — Loader and compiler integration | Not started | 0% | No asset import path classification or constant-binding support yet. |
| M3 — Semantic analysis, lowering, and backend parity | Not started | 0% | No checker/lowering/backend support yet. |
| M4 — LSP and workspace graph parity | Not started | 0% | Workspace/import tooling still assumes `.fuse` module targets. |
| M5 — Validation, fixtures, docs, and release hardening | Not started | 0% | Tests/examples/docs updates not landed. |

### Recommended scope call

`0.9.7` should stay focused on asset-import work plus the minimum parity/docs/test hardening needed
to ship it safely. Do not stack unrelated release items onto this patch.

## Objective

Ship a small, strict, non-plugin import extension that lets FUSE programs import checked
repository assets from `.md` and `.json` files without weakening the current module model.

The feature should feel like an extension of the existing import system, not a new general-purpose
file I/O mechanism.

---

## Current language state

### What exists today

- Imports are currently module-oriented only.
  The parser accepts `import X`, `import X from "..."`, `import {A, B} from "..."`, and
  `import X as Y from "..."`.
- Loader resolution assumes imported files are FUSE modules:
  - missing extensions default to `.fuse`
  - loaded files are read as text and passed through `parse_source(...)`
  - exports are derived from top-level FUSE declarations only
- The only non-file import special-case is the virtual `std.Error` module.
- Runtime already has adjacent building blocks:
  - `json.encode` / `json.decode`
  - `asset(path)` for public URL lookup
  - `svg.inline(path)` for safe textual asset loading
- Tooling assumes imports target modules:
  - semantic import collection records module or item imports only
  - LSP workspace relinking resolves import targets as `.fuse` modules
  - import navigation/code actions are shaped around module exports

### Gaps relevant to `.md` / `.json`

- There is no import path classifier for asset files.
- There is no AST or semantic representation for imported constant values.
- There is no first-class static `Value` or `Json` type in the checker.
  Current `json.decode(...)` behavior is effectively dynamic from the checker’s perspective.
- Import diagnostics, indexing, and navigation all assume the target is a FUSE source file.

### Constraint summary

The implementation has to preserve FUSE’s current identity:

- strict, deterministic loading
- no runtime plugin system
- no arbitrary compile-time extensibility hooks
- parity across AST, native, CLI, and LSP flows

## Proposed `0.9.7` contract

### Supported forms

`0.9.7` should support single-binding asset imports only:

```fuse
import Docs from "./README.md"
import SeedData from "./seed.json"
import Policy from "root:content/policy.md"
import AuthFixture from "dep:Fixtures/auth/login.json"
```

### Binding semantics

- `.md` imports bind a local immutable `String` value containing the UTF-8 file contents exactly as
  authored.
- `.json` imports bind a local immutable value using the same runtime shape produced by
  `json.decode(text)`.
- Imported asset bindings are values, not modules.
- Asset imports do not create a namespace and do not expose named exports.

### Explicit `0.9.7` restrictions

To keep the patch scope tight:

- no named imports from asset files
  - reject `import {x} from "./data.json"`
- no bare module-name asset imports
  - reject `import Docs` for non-`.fuse` assets
- no automatic Markdown-to-HTML conversion
  - `.md` imports are raw strings only
- no generalized TOML/YAML/text loader family yet
- no user-defined import loaders or macros

### Typing strategy

- Markdown imports should be statically typed as `String`.
- JSON imports should use the same semantic posture as current `json.decode(...)`:
  useful at runtime, intentionally conservative in static typing.
- If a small dedicated `Json`/`Value` type can be introduced without broad surface-area churn, it is
  acceptable as an implementation refinement; it is not required to ship the feature.

### Diagnostics expectations

The compiler should report path-attached diagnostics for:

- missing asset file
- unreadable file
- invalid UTF-8 for `.md` / `.json`
- invalid JSON syntax
- unsupported import form for asset paths
- unsupported asset extension in import position

## Milestones

### M1 — Semantics and surface freeze

Goal: lock the user-facing contract before changing loaders or tooling.

Deliverables:

- Define asset import classification by extension: `.fuse`, `.md`, `.json`.
- Freeze the `0.9.7` grammar/semantic rule that only `import Name from "path.ext"` is supported for
  asset files.
- Specify value semantics:
  - `.md` -> `String`
  - `.json` -> decoded runtime value
- Specify diagnostics for rejected forms and malformed files.
- Record the non-goals above in spec/doc updates during implementation.

Exit criteria:

- no unresolved syntax ambiguity with existing module imports
- no open question about whether asset imports are values or modules

### M2 — Loader and compiler integration

Goal: make the compiler resolve and bind asset imports deterministically.

Work items:

- Add import-target classification in the loader before FUSE parsing.
- Extend loader bookkeeping so imports can bind either:
  - module links, or
  - constant asset values
- Keep current `dep:` and `root:` resolution behavior, but allow `.md` / `.json` targets.
- Reuse the strict path normalization posture already used by existing import resolution.
- Parse JSON once during load and store the decoded constant form.
- Read Markdown once during load and store the raw string constant.
- Reject named-import and alias/module-namespace forms for asset targets in this release.

Exit criteria:

- asset imports participate in ordinary check/run/build flows
- repeated imports of the same asset share normalized-path identity
- invalid asset content fails during load/check, not deep at runtime

### M3 — Semantic analysis, lowering, and backend parity

Goal: make imported assets behave like ordinary immutable values everywhere.

Work items:

- Extend symbol collection so asset imports register value bindings rather than module aliases.
- Ensure the checker understands:
  - markdown imports as `String`
  - JSON imports with the chosen conservative type strategy
- Thread asset constants through lowering/canonical forms so AST and native backends observe the
  same values.
- Keep runtime behavior deterministic: importing an asset should not require an additional runtime
  file read after compilation/loading succeeds.

Exit criteria:

- AST and native backends produce identical observable results for imported `.md` / `.json`
- imported assets work in `let`, `return`, config defaults, JSON encoding, and route responses

### M4 — LSP and workspace graph parity

Goal: keep editor behavior aligned with compiler behavior.

Work items:

- Update workspace import-target resolution so `.md` / `.json` paths resolve without forcing
  `.fuse`.
- Ensure incremental relinking invalidates dependents when imported asset files change.
- Support import-path navigation for asset imports.
- Preserve organize-imports and import-related code actions without generating invalid named-import
  suggestions for asset files.
- Keep large-workspace behavior bounded; asset files should not trigger full reindexing when a
  narrow relink is enough.

Exit criteria:

- editing an imported asset updates diagnostics/navigation in the owning FUSE file
- no regression in existing `.fuse`, `dep:`, `root:`, or `std.Error` import flows

### M5 — Validation, fixtures, docs, and release hardening

Goal: land the feature with tests and documentation at the same quality bar as the rest of `0.9.x`.

Work items:

- Add parser/semantic fixtures for accepted and rejected asset import forms.
- Add compiler/load tests for:
  - local relative imports
  - `root:` imports
  - `dep:` imports
  - malformed JSON
  - missing files
  - duplicate imported names
- Add AST/native parity tests covering identical runtime values.
- Add LSP navigation and incremental-workspace tests for asset imports.
- Update docs together:
  - `README.md`
  - `spec/fls.md`
  - `spec/runtime.md`
  - `governance/scope.md` if scope wording changes
  - `guides/reference.md` via `scripts/generate_guide_docs.sh`
- Add at least one example for Markdown import and one for JSON import.

Exit criteria:

- feature is fully documented and example-backed
- no backend or LSP parity gaps remain open for the shipped contract

---

## Recommended sequencing

1. M1 semantics freeze
2. M2 loader/compiler binding work
3. M3 checker/lowering/backend parity
4. M4 LSP parity
5. M5 docs/examples/release validation

This ordering keeps the highest-risk decisions early: import classification, value-vs-module
semantics, and the JSON typing posture.

---

## Release guardrails for `0.9.7`

This should ship only if all of the following are true:

- no existing `.fuse` import form changes meaning
- asset imports remain deterministic and repository-local
- JSON behavior is clearly documented as matching current `json.decode(...)` runtime shape
- LSP import resolution matches CLI/compiler resolution
- docs and examples are updated in the same change window as implementation

If those guardrails cannot be met within patch-release scope, the fallback is to ship Markdown-only
imports in `0.9.7` and defer JSON imports to the next minor planning window.
