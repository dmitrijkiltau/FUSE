# 0.9.3 Patch Plan

Status: Draft
Release line: 0.9.x (patch-safe)

## Objective

Advance multi-package/dependency workflow ergonomics for publishable repositories without changing
core language/runtime semantics.

## Scope guardrails

- Additive CLI/dependency workflow only.
- Existing `fuse.toml` and `fuse.lock` behavior must remain backward compatible.
- Keep transitive dependency resolution and conflict diagnostics deterministic.

## Milestones

### M3.1 Explicit lock maintenance command

Deliverables:

- Add a dedicated lock maintenance command surface (for example `fuse deps lock`).
- Support check-only and update modes for CI and local workflows.

Acceptance criteria:

- Command is documented and covered by integration tests.
- Check mode fails deterministically on lock drift.

### M3.2 Frozen lockfile mode

Deliverables:

- Add a frozen mode for `check/run/build/test` that refuses lockfile mutation.
- Return structured diagnostics with clear remediation hints.

Acceptance criteria:

- Frozen mode is test-covered for path and git dependency changes.
- Existing non-frozen behavior is unchanged.

### M3.3 Offline dependency mode

Deliverables:

- Add offline resolver mode that uses only `fuse.lock` + local `.fuse/deps` cache.
- Emit clear diagnostics when missing cached revisions prevent resolution.

Acceptance criteria:

- Offline mode tests pass for cache hit/miss scenarios.
- No implicit network fallback occurs in offline mode.

### M3.4 Dependency doctor/graph diagnostics

Deliverables:

- Add a dependency graph/doctor command for conflict, cycle, and unresolved-source debugging.
- Provide machine-readable diagnostic codes for CI tooling.

Acceptance criteria:

- Graph output is deterministic for unchanged manifests.
- Doctor command detects known conflict/cycle fixtures.

### M3.5 Publish-readiness checks for workspace repos

Deliverables:

- Add workspace publish-readiness checks (manifest consistency, lock state, dependency sanity).
- Integrate into release preflight as optional checks for multi-package repos.

Acceptance criteria:

- Checks run cleanly on current repo layout.
- Failure output is actionable and includes fix hints.

## Required ship gates

- `./scripts/cargo_env.sh cargo test -p fuse --test project_cli`
- `./scripts/cargo_env.sh cargo test -p fusec --test dep_resolution`
- `./scripts/release_smoke.sh`
- `./scripts/release_preflight.sh 0.9.3 --skip-bench`

## Cut line (must ship)

- M3.1
- M3.2
- M3.5

## Defer-first items (if schedule slips)

- M3.3 if offline UX is not robust enough for CI usage.
- M3.4 advanced graph visualization beyond textual deterministic output.
