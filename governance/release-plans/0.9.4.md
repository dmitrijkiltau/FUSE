# 0.9.4 Patch Plan

Status: Draft
Release line: 0.9.x (patch-safe)

## Objective

Strengthen release artifact integrity and provenance automation in CI while preserving the current
artifact contract and deterministic metadata behavior.

## Scope guardrails

- No artifact naming contract breaks (`fuse-cli-*`, `fuse-aot-*`, `fuse-vscode-*` remain stable).
- Keep checksum and metadata generation behavior backward compatible.
- Add verification layers; do not remove existing release gates.

## Milestones

### M4.1 Artifact signing pipeline

Deliverables:

- Add automated signing for release artifacts and/or checksum manifest.
- Emit detached signature files in `dist/` with deterministic naming.

Acceptance criteria:

- Signatures are generated in release workflow runs.
- Verification command/script succeeds for generated artifacts.

### M4.2 SBOM generation

Deliverables:

- Generate SBOMs for CLI, AOT, and VSIX release payloads.
- Publish SBOM files as release assets.

Acceptance criteria:

- SBOM generation is reproducible in CI.
- SBOM files are listed in release metadata documentation.

### M4.3 Provenance attestations

Deliverables:

- Add CI-produced provenance attestations tied to release artifacts.
- Include commit/tag/workflow identity fields needed for auditing.

Acceptance criteria:

- Attestation generation is gated on tagged release publishing.
- Attestation files are attached alongside release assets.

### M4.4 Workflow verification hardening

Deliverables:

- Extend release workflow verification stage to validate signatures/SBOM/provenance outputs.
- Fail publish pipeline on verification mismatch.

Acceptance criteria:

- `release-artifacts.yml` fails on tampered or missing integrity artifacts.
- Dry-run mode validates build + integrity generation without publish.

### M4.5 Operator documentation and runbook updates

Deliverables:

- Update `README.md` and `ops/RELEASE.md` with end-user verification commands.
- Update contract notes in `ops/AOT_RELEASE_CONTRACT.md` if scope changes.

Acceptance criteria:

- Documentation includes clear verification steps for downloaded assets.
- Release checklist references new integrity outputs and checks.

## Required ship gates

- `./scripts/release_smoke.sh`
- `./scripts/package_release.sh --release --skip-container`
- `.github/workflows/release-artifacts.yml` dry-run validation
- `./scripts/release_preflight.sh 0.9.4`

## Cut line (must ship)

- M4.1
- M4.4
- M4.5

## Defer-first items (if schedule slips)

- M4.2 or M4.3 only if rollout complexity threatens release stability.
