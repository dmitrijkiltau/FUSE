# 0.9.9 plan: contract audit and tooling polish

## Release status

- Decision: reserve as a stabilization release after `0.9.7` and `0.9.8` land.
- Objective: remove remaining contract ambiguity before `1.0.0`.
- Current progress: `M1` remains active and `M2` is complete with documented machine-readable
  coverage across loader, runtime, and wrapper failure classes.

| Milestone | Status | Progress | Notes |
|---|---|---:|---|
| M1 — Spec/guide audit | In progress | 25% | Ownership map captured; initial README/reference summary drift (`component`, `transaction`) identified and being aligned. |
| M2 — Diagnostic consistency | Completed | 100% | High-value loader, runtime, and wrapper failure classes now have stable documented JSON/LSP/CLI diagnostic surfaces. |
| M3 — LSP quality pass | In progress | 80% | Initial audit complete; module-alias navigation/refactor/signature/completion/workspace-symbol flows are now pinned in both focused regressions and raw wire-level UX smoke coverage. |
| M4 — Example and reference-service polish | Not started | 0% | Ensure flagship examples exercise the intended stable contract. |
| M5 — 1.0 blocker review | Not started | 0% | Produce a clean blocker list for the final pre-1.0 patch. |

## Why `0.9.9`

After feature-completion patches, the biggest risk to a weak `1.0.0` is ambiguity rather than raw
missing capability. This release should reduce surprise by tightening docs, diagnostics, examples,
and editor behavior.

## Scope

### In scope

- documentation/spec/reference alignment
- high-value structured diagnostic coverage
- LSP correctness polish for stabilized features
- example/reference-service cleanup to match the documented contract
- explicit `1.0.0` blocker inventory

### Out of scope

- major new language constructs
- breaking syntax or semantic changes
- large runtime subsystems that would deserve their own release

## Milestones

### M1 — Spec/guide audit

Goal: ensure the published contract says one thing everywhere.

Deliverables:

- audit of `README.md`, `spec/fls.md`, `spec/runtime.md`, and `guides/reference.md`
- removal of stale wording from earlier `0.9.x` patches
- explicit coverage map for stable language/runtime/tooling surfaces

Exit criteria:

- no known contract-facing contradictions remain

Initial audit snapshot:

| Surface | Owning document | Current M1 note |
|---|---|---|
| Onboarding, install, command routing, doc map | `README.md` | Front-door summary is being tightened to match the stabilized declaration surface. |
| Syntax, grammar, imports, static typing, compile-time restrictions | `spec/fls.md` | Treat as the canonical language contract for `component`, `transaction`, capabilities, typed errors, and import forms. |
| Validation/JSON/config/CLI/HTTP binding, runtime errors, DB/concurrency, AOT | `spec/runtime.md` | Treat as the canonical runtime contract; no ownership drift found in the initial pass. |
| Generated developer reference | `guides/reference.md` via `scripts/generate_guide_docs.sh` | Summary section had stale omissions and must stay derived from spec/runtime/scope rather than hand-edited. |
| Scope, priorities, non-goals | `governance/scope.md` | Planning-only boundary doc; use it to describe intent, not to redefine semantics. |

Initial findings:

- the README language summary omitted `component` from the stable declaration list
- the generated reference summary omitted `component` from top-level declarations and `transaction`
  from core statements
- next M1 pass should audit README command/runtime wording against the normative `spec/fls.md` and
  `spec/runtime.md` text, then verify the same posture in `guides/reference.md`

### M2 — Diagnostic consistency

Goal: make important compiler/runtime failures easier to automate and migrate.

Deliverables:

- identify remaining high-value diagnostics lacking machine-readable codes or actionable wording
- improve structured diagnostic consistency where it does not alter behavior
- preserve existing patch-level compatibility

Exit criteria:

- major user-facing failure classes have stable, documented diagnostics

Initial M2 focus:

- prioritize load-time import/asset failures because they surface in CLI JSON diagnostics and LSP
  diagnostics but previously had inconsistent or missing codes
- normalize high-value loader failures under stable families `FUSE_IMPORT_*` and `FUSE_ASSET_*`
- keep messages actionable by naming the failing import form, asset path, or available dependency
  alternatives

Current M2 additions:

- dependency graph cycle diagnostics are now part of the machine-readable contract via
  `FUSE_DEP_CYCLE`
- derived-type loader failures now use stable `FUSE_TYPE_DERIVE_*` codes
- duplicate exported-symbol loader failures now use `FUSE_SYMBOL_DUPLICATE`
- runtime execution envelopes now classify common failures under stable `runtime_*` codes instead
  of collapsing everything to generic `runtime_error`
- config/env decode failures now classify under `runtime_config_decode`, and direct AOT fatal
  envelopes are covered for matching config/operator runtime message taxonomy
- structured config/env binding failures are explicitly tracked as validation payloads rather than
  generic uncoded runtime text
- wrapper-side JSON `cli_message` failures now carry stable codes for common command/backend/
  manifest/subcommand/file/dev failure classes

M2 completion snapshot:

- loader/import/asset failure families are normalized under stable `FUSE_IMPORT_*`,
  `FUSE_ASSET_*`, `FUSE_DEP_CYCLE`, `FUSE_TYPE_DERIVE_*`, and `FUSE_SYMBOL_DUPLICATE` codes
- CLI/LSP JSON diagnostics consistently carry `code` for the high-value compiler/loader failures
  that users are most likely to automate against
- runtime execution failures exposed through JSON validation envelopes now classify common
  operator/task/type/config failures under stable `runtime_*` codes, while structured config/env
  binding failures remain explicitly field-addressed validation payloads
- wrapper-side JSON `cli_message` output now carries stable codes for common command/backend/
  manifest/subcommand/file/dev/setup failures instead of relying on free-form text alone
- direct AOT fatal envelopes are covered for matching runtime message taxonomy on operator-path and
  config/env decode failures

Remaining non-blocking follow-up:

- keep adding codes for niche or newly introduced failures as features evolve, but the current
  documented contract now covers the major user-facing classes required for `1.0.0` automation and
  migration guidance

### M3 — LSP quality pass

Goal: make editor workflows trustworthy enough for `1.0.0` day-one use.

Deliverables:

- import/navigation/code-action reliability audit
- completion/signature relevance cleanup where existing behavior is noisy
- incremental-workspace regression additions for stabilized features

Exit criteria:

- no known high-frequency LSP correctness issues remain open

Initial audit snapshot:

| Surface | Current M3 note |
|---|---|
| Navigation and rename (`definition`, `references`, `prepareRename`, `rename`, call hierarchy) | Core cross-module flows are covered, but named-import rename propagation was not yet pinned by regression coverage. |
| Code actions (`quickfix`, `source.organizeImports`) | Existing import/config/capability/HTML migration fixes have focused coverage and no immediate contract drift was found. |
| Completion and signature help | Current ranking/active-parameter coverage is solid enough for this pass; keep watching large-graph noise rather than semantics. |
| Incremental workspace cache/relink | Incremental manifest/import/relink scenarios are already exercised and should stay regression-gated while M3 focuses on correctness polish. |

Current M3 additions:

- added a focused regression for cross-module rename so renaming an exported symbol also rewrites
  named-import bindings in dependent modules, not just the definition and callsites
- added focused signature-help coverage for calls routed through imported module aliases so
  `util.fn(...)`-style editor flows stay pinned alongside direct and builtin call coverage
- fixed definition navigation on module-alias receivers so jumping on `util` in `util.greet(...)`
  opens the imported module file instead of bouncing back to the local import binding
- added module-alias receiver reference/rename coverage so `prepareRename`, `rename`, and
  `references` on `util` in `util.greet(...)` now track the local module binding and its use sites
- enriched module-alias receiver hover so `util` in `util.greet(...)` shows the imported module
  target plus exported symbols instead of only the local binding stub
- extended the raw `lsp_ux` wire-level smoke test so root/dep module aliases are exercised for
  receiver-token `definition`, `hover`, `references`, and local alias rename behavior over JSON-RPC
- extended the raw `lsp_ux` wire-level smoke test further so module-alias receiver
  `prepareRename` is also covered over JSON-RPC for both root and dependency aliases
- extended the raw `lsp_ux` wire-level smoke test again so module-alias receiver member completion
  and `workspace/symbol` lookup for imported aliases are both exercised over JSON-RPC
