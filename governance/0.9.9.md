# 0.9.9 plan: contract audit and tooling polish

## Release status

- Decision: reserve as a stabilization release after `0.9.7` and `0.9.8` land.
- Objective: remove remaining contract ambiguity before `1.0.0`.
- Current progress: `M1` remains active and `M2` has started with loader/import diagnostic-code
  normalization for JSON/LSP consumers.

| Milestone | Status | Progress | Notes |
|---|---|---:|---|
| M1 — Spec/guide audit | In progress | 25% | Ownership map captured; initial README/reference summary drift (`component`, `transaction`) identified and being aligned. |
| M2 — Diagnostic consistency | In progress | 45% | Stabilizing high-value loader/import/type/dependency diagnostic codes and documenting their intended machine-readable surface. |
| M3 — LSP quality pass | Not started | 0% | Focus on import/navigation/code-action correctness on stabilized language features. |
| M4 — Example and reference-service polish | Not started | 0% | Ensure flagship examples exercise the intended stable contract. |
| M5 — 1.0 blocker review | Not started | 0% | Produce a clean blocker list for the final pre-1.0 patch. |

## Why `0.9.9`

After feature-completion patches, the biggest risk to a weak `1.0.0` is ambiguity rather than raw
missing capability. This release should reduce surprise by tightening docs, diagnostics, examples,
and editor behavior.

## Scope

### In scope

- documentation/spec/reference alignment
- high-value structured diagnostic coverage
- LSP correctness polish for stabilized features
- example/reference-service cleanup to match the documented contract
- explicit `1.0.0` blocker inventory

### Out of scope

- major new language constructs
- breaking syntax or semantic changes
- large runtime subsystems that would deserve their own release

## Milestones

### M1 — Spec/guide audit

Goal: ensure the published contract says one thing everywhere.

Deliverables:

- audit of `README.md`, `spec/fls.md`, `spec/runtime.md`, and `guides/reference.md`
- removal of stale wording from earlier `0.9.x` patches
- explicit coverage map for stable language/runtime/tooling surfaces

Exit criteria:

- no known contract-facing contradictions remain

Initial audit snapshot:

| Surface | Owning document | Current M1 note |
|---|---|---|
| Onboarding, install, command routing, doc map | `README.md` | Front-door summary is being tightened to match the stabilized declaration surface. |
| Syntax, grammar, imports, static typing, compile-time restrictions | `spec/fls.md` | Treat as the canonical language contract for `component`, `transaction`, capabilities, typed errors, and import forms. |
| Validation/JSON/config/CLI/HTTP binding, runtime errors, DB/concurrency, AOT | `spec/runtime.md` | Treat as the canonical runtime contract; no ownership drift found in the initial pass. |
| Generated developer reference | `guides/reference.md` via `scripts/generate_guide_docs.sh` | Summary section had stale omissions and must stay derived from spec/runtime/scope rather than hand-edited. |
| Scope, priorities, non-goals | `governance/scope.md` | Planning-only boundary doc; use it to describe intent, not to redefine semantics. |

Initial findings:

- the README language summary omitted `component` from the stable declaration list
- the generated reference summary omitted `component` from top-level declarations and `transaction`
  from core statements
- next M1 pass should audit README command/runtime wording against the normative `spec/fls.md` and
  `spec/runtime.md` text, then verify the same posture in `guides/reference.md`

### M2 — Diagnostic consistency

Goal: make important compiler/runtime failures easier to automate and migrate.

Deliverables:

- identify remaining high-value diagnostics lacking machine-readable codes or actionable wording
- improve structured diagnostic consistency where it does not alter behavior
- preserve existing patch-level compatibility

Exit criteria:

- major user-facing failure classes have stable, documented diagnostics

Initial M2 focus:

- prioritize load-time import/asset failures because they surface in CLI JSON diagnostics and LSP
  diagnostics but previously had inconsistent or missing codes
- normalize high-value loader failures under stable families `FUSE_IMPORT_*` and `FUSE_ASSET_*`
- keep messages actionable by naming the failing import form, asset path, or available dependency
  alternatives

Current M2 additions:

- dependency graph cycle diagnostics are now part of the machine-readable contract via
  `FUSE_DEP_CYCLE`
- derived-type loader failures now use stable `FUSE_TYPE_DERIVE_*` codes
- duplicate exported-symbol loader failures now use `FUSE_SYMBOL_DUPLICATE`

### M3 — LSP quality pass

Goal: make editor workflows trustworthy enough for `1.0.0` day-one use.

Deliverables:

- import/navigation/code-action reliability audit
- completion/signature relevance cleanup where existing behavior is noisy
- incremental-workspace regression additions for stabilized features

Exit criteria:

- no known high-frequency LSP correctness issues remain open

### M4 — Example and reference-service polish

Goal: ensure canonical examples model the intended stable posture.

Deliverables:

- refresh examples to prefer stable, documented idioms
- align `examples/reference-service/` with the latest recommended workflows
- close doc/example drift

Exit criteria:

- examples and docs tell the same story

### M5 — 1.0 blocker review

Goal: convert broad readiness concerns into a bounded final blocker list.

Deliverables:

- explicit blocker categories: semantics, runtime parity, tooling, docs, release ops
- recommendation for whether `0.9.10` is enough before `1.0.0`

Exit criteria:

- `1.0.0` readiness is measurable, not aspirational
