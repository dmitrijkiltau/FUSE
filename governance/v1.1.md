## Release character

v1.1 is an additive minor release under the `1.x` compatibility guarantee.

The primary objective is a closed-set, static-dispatch interface contract system that removes repeated structural contracts across package boundaries without introducing runtime polymorphism.

The editor objective for this slice is limited to the interface launch surface required to make the feature usable in normal authoring flows. Broader LSP backlog items remain deferred.

---

## Motivation

At `1.0.x`, FUSE has no way to state that multiple nominal data types share the same behavioral contract. Authors either duplicate function signatures, rely on naming conventions, or hard-code concrete types at package boundaries.

v1.1 addresses that gap with a compile-time-only `interface` / `impl` model. The implementation relationship is checked in the frontend, concrete dispatch is resolved before backend lowering, and interfaces have no runtime representation.

The feature must remain consistent with the identity charter. In particular, v1.1 does not introduce:

- runtime dispatch or vtables
- runtime interface checks or downcasts
- open implementation sets or blanket implementations
- default interface methods
- interface inheritance or composition
- operator overloading through interfaces

---

## Delivered surface

The shipped v1.1 surface is:

- top-level `interface` declarations
- top-level `impl Interface for Type` blocks
- `Self` support inside interface member signatures, impl member signatures, and impl bodies
- interface export/import at the same top-level tier as types and enums
- static resolution for concrete `value.method()` and `Type.method()` calls through impl members
- frontend desugaring of impl dispatch to internal direct-call targets before interpreter/native lowering

The first slice intentionally excludes:

- `where` syntax
- interface-constrained generics or parameters
- interface values or dynamic interface operations

`where` is intentionally not reserved in v1.1 so existing expressions such as `.where(...)` remain valid.

## Grammar surface

The normative grammar lives in `spec/fls.md`. The shipped shape is:

```ebnf
interface_decl ::= "interface" IDENT ":" NEWLINE INDENT interface_member+ DEDENT
interface_member ::= "fn" IDENT "(" param_list? ")" ("->" type_expr ("!" IDENT)?)? NEWLINE
impl_decl ::= "impl" IDENT "for" type_expr ":" NEWLINE INDENT fn_decl+ DEDENT
```

`interface` and `impl` are reserved keywords in v1.1. Identifier-position misuse is diagnosed with `FUSE_RESERVED_KEYWORD` and a rename hint.

---

## Static semantics

The shipped v1.1 rules are:

1. `interface` defines a named contract, not a runtime type.
2. Interface names export and import like other top-level declarations.
3. `impl` blocks are checked at declaration time and are not importable declarations themselves.
4. An impl target must be a nominal data type (`type` or `enum`, including imported equivalents).
5. Interface names are not valid in ordinary type positions. Misuse is diagnosed as `FUSE_INTERFACE_NOT_A_TYPE`.
6. Each `(interface, concrete type)` pair may have at most one impl in the visible package graph.
7. Each impl must satisfy every interface member, after substituting `Self` with the concrete impl target.
8. Orphan rules apply: an impl is only legal when the current package owns the interface or the concrete target type.
9. Instance members receive an implicit immutable `self: ConcreteType` receiver.
10. Associated members have no implicit receiver.
11. `Self` is only valid inside interface/impl signatures and impl bodies; it is not a general-purpose type keyword.
12. Interface dispatch is resolved in the frontend and lowered as direct calls. Backends do not gain interface-specific runtime behavior.

## Example

```fuse
interface Codec:
  fn encode(format: String) -> String
  fn decode(text: String) -> Self

type Note:
  text: String

impl Codec for Note:
  fn encode(format: String) -> String:
    return self.text

  fn decode(text: String) -> Note:
    return Note(text: text)

fn render(note: Note) -> String:
  return note.encode("json")

fn parse(text: String) -> Note:
  return Note.decode(text)
```

This is the full v1.1 model. There is no `where` clause in this slice.

## Diagnostics reserved and shipped in v1.1

- `FUSE_RESERVED_KEYWORD`: `interface` or `impl` used where an identifier is expected
- `FUSE_INTERFACE_NOT_A_TYPE`: interface name used in an ordinary type position
- `FUSE_IMPL_DUPLICATE`: duplicate impl for the same `(interface, type)` pair
- `FUSE_IMPL_INCOMPLETE`: impl does not provide all required members
- `FUSE_IMPL_SIGNATURE_MISMATCH`: impl member signature does not match the interface member after `Self` substitution
- `FUSE_IMPL_ORPHAN`: impl violates package ownership rules
- `FUSE_INTERFACE_DYNAMIC_USE`: attempt to use an interface as a runtime value or dynamic target

## LSP launch surface

The v1.1 editor surface shipped with interface support includes:

- diagnostics for the reserved v1.1 interface/impl codes
- go-to-definition from an `impl Interface for Type` header to the `interface` declaration
- workspace symbols listing interfaces under the existing `type:` tier
- completion inside impl bodies for missing interface member stubs
- code action to generate an impl skeleton for a selected interface/type pair
- signature help that prefers concrete impl members over interface declarations when both are applicable

This slice does not include constrained-parameter hover because `where` is deferred.

---

## Deferred follow-on work

The following items are intentionally out of scope for the shipped v1.1 slice:

- `where` syntax and interface-constrained parameters/generics
- hover for constrained parameters
- rename through re-export chains
- builtin hover documentation sourced from `spec/runtime.md`
- inlay hints for propagated error domains

These remain follow-on RFC or implementation work and are not required for the v1.1 release gate.

## Compatibility guarantee

All programs valid on `1.0.x` remain valid on `1.1.x`, except for the ordinary reserved-keyword tightening around `interface` and `impl` in identifier positions.

`where` remains unreserved in v1.1. Existing source that uses `.where(...)` or `where` as an identifier outside the new keyword surface remains source-compatible.

Because interface dispatch is fully resolved in the frontend, v1.1 does not introduce backend-specific runtime divergence.
